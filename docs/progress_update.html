Recap: Key Findings So Far

Model Performance

  • Small local models (4B, 32B) underperform — too poor to justify over API
  • Multi-agent teams > single-agent self-improving setups
  • Models show divergent strategies: Kimi uses fewer tool calls + longer thinking; others prefer more tool calls + shorter generation

Cost & Latency

  • Apples-to-oranges costing: Single-user local appears costlier than API, but API pricing amortizes multi-user GPU concurrency
  • Token-based costing is flawed: Poor models generate many useless tokens — normalizing by tokens is misleading
  • E2E latency is king: Primary user metric. P99 fluctuates wildly by token count and server congestion; tool calls are near-zero overhead (CPU-only)

Cost Model Design

Scenario 1: API Models

cost = input_tokens × input_price + output_tokens × output_price

Scenarios 2-4: Local Models

cost = total_per_hour × gpu_seconds / 3600
total_per_hour = CapEx/hr + OpEx/hr = $2.92/hr (2×H100)

Unified Metrics Framework

Single runner for all datasets. Swap dataset, keep same metrics and output format.

Per Example (one task)

  • total_input_tokens, total_output_tokens
  • tool_call_count, num_requests
  • e2e_latency_s
  • avg/max input_tokens_per_request
  • total_prefill_time_s, total_decode_time_s

Per Request (one LLM call)

  • prefill_time_s (TTFT)
  • decode_time_s
  • tpot_s (time per output token)
  • output_throughput_tok_s
  • input_tokens, output_tokens

Dataset-Level Aggregation

  • p50/p99 e2e_latency, prefill, decode, tpot
  • examples_per_second
  • avg tool_calls, avg tokens per request

Hardware

  • gpu_name, gpu_count
  • gpu_utilization_pct, gpu_memory_used_mb
  • cpu_name, cpu_utilization_pct
Output: metadata.json + metrics.json + per_example_results.jsonl + per_request_details.jsonl
